Txt-code 0.2 — Full Breakdown

  ---
  What Is Txt-code?

  Txt-code is a programming language you built from scratch in Rust. It is designed for:

  - Security-first automation — you can control exactly what a script is allowed to do
  - AI-safe scripting — structured, predictable output for AI agents to read and act on
  - Cyber orchestration — safely running security tools and network operations with permission controls
  - Audit trails — every action is logged so you always know what ran and why

  Think of it like Python or Bash, but with a built-in permission system, policy engine, and audit log baked into the runtime itself.

  ---
  The Language Syntax

  Txt-code uses an arrow-based style that reads almost like English:

  store → name → "Alice"        # create a variable
  print → "Hello, " + name      # print it

  define → greet → (name)       # define a function
    return → "Hello, " + name
  end

  if → age > 18                 # if statement
    print → "Adult"
  else
    print → "Minor"
  end

  Two styles work — arrow → or space-based (both accepted):
  store → x → 42      # arrow style (preferred)
  store x 42          # space style (shorthand)

  ---
  How the Language Is Built — Layer by Layer

  Source code (.tc file)
          ↓
     [ LEXER ]        Breaks text into tokens (words, symbols, numbers)
          ↓
     [ PARSER ]       Builds a tree structure (AST) from those tokens
          ↓
     [ TYPE CHECKER ] Checks types make sense (like a spell-checker for types)
          ↓
     [ VALIDATOR ]    Checks semantic rules (e.g. no circular imports)
          ↓
     [ VM / COMPILER ] Executes the tree OR compiles it to bytecode
          ↓
     [ STDLIB ]       Provides built-in functions (print, http_get, read_file...)

  1. Lexer (src/lexer/)

  Reads raw text and produces tokens:
  "store → x → 42"
  → [KEYWORD:store] [ARROW] [IDENTIFIER:x] [ARROW] [INTEGER:42]

  2. Parser (src/parser/)

  Turns tokens into an Abstract Syntax Tree (AST) — a tree of nodes:
  Assignment {
    pattern: Identifier("x"),
    value:   Literal(Integer(42))
  }

  3. Type Checker (src/typecheck/)

  Uses Hindley-Milner inference — it figures out types automatically without you having to write them everywhere. You can annotate types if you want:
  store → x: int → 42          # optional annotation
  store → y → 42               # inferred automatically

  4. Compiler (src/compiler/)

  Two outputs:
  - AST VM path — the tree is walked directly (used by txtcode run)
  - Bytecode path — the tree is compiled to instructions (used by txtcode compile)

  ---
  The Two Execution Engines

  This is the most important technical detail to understand:

  Engine 1: AST VM (Production)

  File: src/runtime/vm.rs

  This is the real, safe engine. When you run txtcode run script.tc:
  - Walks the AST node by node
  - Checks every permission before every sensitive operation
  - Logs every action to the audit trail
  - Enforces policy rules (rate limits, timeouts)
  - Enforces allowed/forbidden capability declarations

  txtcode run script.tc   →   AST VM   →   Full security, audit, permissions
  txtcode repl            →   AST VM   →   Same guarantees interactively

  Engine 2: Bytecode VM (Experimental)

  File: src/runtime/bytecode_vm.rs

  This is a stack-based machine — like a mini CPU with a stack of values:
  PUSH 10          stack: [10]
  PUSH 5           stack: [10, 5]
  ADD              stack: [15]       ← pops two, pushes result

  Used by txtcode compile and txtcode bench. In v0.2 it is experimental — it skips permissions and audit. This is the engine we hardened most in this release.

  ---
  The Permission System (The Heart of Security)

  This is what makes Txt-code different from Python or Bash. Every sensitive operation must be permitted.

  How it works — 3 layers:

  Layer 1: Capability Declarations in Code
  define → read_config → (path: string)
    allowed   → ["fs.read"]              # this function CAN read files
    forbidden → ["net.connect", "sys.exec"]  # but CANNOT do networking or exec
    return → read_file(path)
  end

  Layer 2: PermissionChecker Trait
  The VM implements a PermissionChecker Rust trait. Before any net/IO/exec call, the stdlib checks it:
  http_get("example.com")
    → StdLib checks: is net.connect permitted?
    → if denied → RuntimeError immediately, no network call made
    → if allowed → proceed, log to audit trail

  Layer 3: Explicit Grant/Deny API
  vm.grant_permission(PermissionResource::FileSystem("read"), Some("/tmp/*"))
  vm.deny_permission(PermissionResource::Network("connect"), None)

  Permission Resources:

  ┌──────────┬─────────────────────┬────────────────────┐
  │ Resource │       Actions       │      Example       │
  ├──────────┼─────────────────────┼────────────────────┤
  │ fs       │ read, write, delete │ fs.read:/tmp/*     │
  ├──────────┼─────────────────────┼────────────────────┤
  │ net      │ connect, listen     │ net.connect        │
  ├──────────┼─────────────────────┼────────────────────┤
  │ sys      │ exec, env           │ sys.exec           │
  ├──────────┼─────────────────────┼────────────────────┤
  │ process  │ named commands      │ process:[ls, curl] │
  └──────────┴─────────────────────┴────────────────────┘

  Scoped permissions: "fs.read:/tmp/*" means read is allowed only inside /tmp/.

  ---
  The Audit Trail

  Every action that touches a resource is logged automatically:
  [2026-03-07 12:34:56] ALLOWED  net.connect → example.com  [intent: fetch_data]
  [2026-03-07 12:34:57] DENIED   sys.exec    → bash         [policy: safe-mode]
  [2026-03-07 12:34:58] ALLOWED  fs.read     → /tmp/data.txt

  Stored in ~/.txtcode/logs/. You can replay and inspect exactly what a script did.

  ---
  Package Manager

  Works like npm or cargo — you declare dependencies in Txtcode.toml:
  [dependencies]
  http-utils = "1.2.0"
  crypto-tools = "^2.0"

  Commands:
  txtcode package init my-project    # create Txtcode.toml
  txtcode package add http-utils     # add a dependency
  txtcode package install            # install all
  txtcode package update             # update to latest
  txtcode package list               # show status

  Packages are fetched from GitHub releases. A lockfile (Txtcode.lock) records exact versions for reproducibility.

  ---
  What I Changed in the 0.2 Hardening

  Here is every change made, explained simply:

  ---
  Fix 1: NullCoalesce Instruction (Bytecode VM)

  Problem: The ?? operator (null coalescing — "use X, or if null use Y") was silently doing nothing in the bytecode VM. It emitted a Nop (no operation).

  What I did:
  - Added a real NullCoalesce instruction to the instruction set
  - Implemented it in the bytecode VM: pop two values, if the first is null → return the second, else → return the first
  - The compiler now emits NullCoalesce instead of Nop

  Before: value ?? default silently produced nothing in bytecode
  After: works correctly — returns default only when value is null

  ---
  Fix 2: Optional Chaining — Clear Errors Instead of Silent Nothing

  Problem: ?., ?[], ?() (optional chaining — safe access that returns null instead of crashing) emitted Nop in bytecode. So your code appeared to run but silently did nothing.

  What I did:
  - Added named instruction variants: OptionalGetField, OptionalIndex, OptionalCall
  - Made the bytecode VM raise a clear RuntimeError explaining the limitation and pointing to txtcode run as the alternative

  Before: user?.name in bytecode → silent nothing
  After: user?.name in bytecode → clear error: "Optional member access is not yet implemented in the bytecode VM. Use an explicit null check, or run with txtcode run."

  ---
  Fix 3: Permission Wiring Fixed

  Problem: The function call_function_with_combined_traits (which should check permissions when the executor implements PermissionChecker) was passing None for the permission checker due to a Rust borrow-checker
  conflict. Permissions were being skipped.

  What I did:
  - Split the permission check into two steps: check first (immutable borrow), then call (mutable borrow)
  - Categorized functions by their resource:
    - http_get, tcp_connect → requires net.connect
    - read_file, write_file → requires fs.read or fs.write
    - exec, spawn → requires process permission
    - getenv, setenv → requires sys.env
  - Now all sensitive stdlib calls actually check the permission before running

  Before: permission denials were bypassed silently
  After: denied operations raise RuntimeError before the stdlib function even runs

  ---
  Fix 4: Pre-existing Build Error

  Problem: user_config.runtime.quiet was referenced but RuntimeConfig has no quiet field. The project didn't compile.

  What I did: Changed cli.quiet || user_config.runtime.quiet to just cli.quiet (the CLI already has the flag; it just doesn't have a config-file equivalent).
